3) Role-Specific: Presales, QA, Setup, Data

  • Confirm demo flow: what scenarios will be shown and what “success” looks like.
  • Smoke test: login, navigation, key modules load time, critical buttons/actions.
  • End-to-end scenario test: create → update → assign → SLA/automation → close → report.
  • Roles/permissions: validate what each demo user should see and do.
  • Data integrity: dashboards, reports, and KPI numbers reflect the demo data properly.
  • Performance basics: no obvious lag in critical screens; pre-load heavy pages if needed.
  • Backups: screenshot/video fallback, alternate scenario, and “plan B” dataset.
  • Document issues: steps to reproduce + impact + workaround for presales team.
  • Title: clear and specific (e.g., “Case assignment rule fails when priority=High”).
  • Environment: version/build, browser/device, user role.
  • Steps to reproduce: numbered steps.
  • Expected vs Actual: what should happen and what happened.
  • Evidence: screenshots/video, logs if available.
  • Impact: blocks demo / blocks user / minor UI issue.
  • Workaround: if any (helps presales keep moving).

Approach: make the demo look like their real day-to-day workflow.

  • Discover: their main use cases (onboarding, customer service, complaint handling, claims, loan applications, KYC).
  • Configure: relevant fields, stages, SLAs, queues, roles, and dashboards.
  • Data: realistic customers, accounts, cases/tickets, interactions, and outcomes.
  • Scenarios: 2–3 short stories (e.g., “new customer onboarding”, “escalated complaint”, “follow-up call”).
  • Validate: run the scenario end-to-end twice; confirm KPIs update correctly.
  • Materials: a short agenda slide + backup screenshots/video + FAQ on limitations.

Key goal: prospect sees “this fits us” rather than “this is just software features.”

  • Create a simple data dictionary (entities, required fields, relationships).
  • Use consistent naming conventions (customers, cases, products, industries).
  • Include a mix of data to show features: new items, in-progress, escalated, overdue, closed.
  • Ensure numbers support dashboards (enough records to make charts meaningful).
  • Keep data “safe” and fictional (no real personal data).
  • Version the dataset (v1, v2) so you can roll back if something breaks.
  • Reproduce quickly and confirm it’s not user/data/config error.
  • Assess impact: Does it affect the critical demo path?
  • If critical: apply workaround (alternative flow, different record, disable a step) and inform presales.
  • Prepare fallback: screenshots or a short video clip for that portion.
  • Log the bug clearly for product/tech team; after demo, retest and close loop.
  • Clarify the real objective (what problem are they trying to solve?).
  • Check if it can be achieved via configuration, workflow redesign, integration, or a workaround.
  • Document the gap clearly and confirm constraints (time, cost, compliance).
  • Escalate internally for feasibility (roadmap, customization, integration approach).
  • Respond professionally: what’s supported now, what’s possible with effort, and recommended next steps.

5) UI Mockups, Videos, and Presales Materials

I’m comfortable with creating clear visuals and mock screens to communicate workflows. If the tool is Adobe XD or Canva, I can adapt quickly—my approach is to focus on: layout clarity, user flow, labels, and how the screen supports the process being demonstrated.

The goal isn’t perfect UI artistry—it’s to help stakeholders visualize a solution clearly and fast.

  • Clarify the user goal and the key fields/actions needed on the screen.
  • Sketch a rough wireframe first (even on paper) → then build in Canva/XD.
  • Keep it simple: key sections, buttons, and status indicators.
  • Validate with presales consultant: does it match the demo narrative?
  • Export and store with versioning in the presales repository.
  • Standard folder structure by industry/use case (Banking, Insurance, Healthcare, etc.).
  • Consistent naming: [Industry]_[UseCase]_[Version]_[Date].
  • Short clips (60–180 seconds) for specific features + a longer “end-to-end scenario” video.
  • Track what changed per version and retire outdated videos.
  • Keep a simple index sheet: title, purpose, last updated, owner, link.
  • Starts with the prospect’s pain points and desired outcomes.
  • Maps requirements → solution approach clearly (fit-gap).
  • Uses simple diagrams/workflows, not only feature lists.
  • States assumptions, scope, timeline, and responsibilities.
  • Clean structure and consistent formatting; easy to scan.
